Which MCP specification should teams implement?

Use the stable specification as the production baseline, track roadmap work separately, and make version support explicit.

Reviewed July 18, 2026Confidence: HighPrimary sources

IN 30 SECONDS

The official MCP specification currently identifies version 2025-11-25 as latest. Production systems should declare supported protocol versions, test compatibility, and avoid treating proposals or roadmap items as stable requirements.

Current production baseline

The authoritative MCP specification page identifies 2025-11-25 as the latest stable version. That version should anchor compatibility, testing, and security review until a newer stable specification is officially published.

What version discipline looks like

  • Record the protocol version supported by every client and server.
  • Test version negotiation and failure behavior.
  • Separate experimental features from production dependencies.
  • Track deprecations and migration work in release management.
  • Re-run security and interoperability tests when the stable version changes.

What not to do

Do not implement roadmap language as if it were final. Do not assume every MCP server supports the same transport, authorization pattern, or optional capability. Do not hide version dependencies inside an agent prompt or an undocumented connector.

What to watch next

Monitor the official specification, changelog, development roadmap, and security guidance. When a new stable version appears, evaluate the differences against your identity, authorization, tasks, tool schemas, logging, and client compatibility.
